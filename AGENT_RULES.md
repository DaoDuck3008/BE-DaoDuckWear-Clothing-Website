# Backend Agent Rules - DaoDuckWear

Follow these rules strictly when developing the backend for DaoDuckWear.

## 1. General Standards

- **Language**: All rules, documentation, and code comments must be in **English**.
- **Naming Conventions**:
  - **Files**: Use `camelCase` (e.g., `productsService.ts`, `authController.ts`).
  - **Functions/Methods**: Use `camelCase` (e.g., `getProductById`, `createNewUser`).
  - **Classes**: Use `PascalCase` (e.g., `ProductsService`, `GlobalExceptionFilter`).
  - **Variables**: Use `camelCase`.

## 2. Project Architecture (NestJS)

- **Modular Structure**: Follow the existing module-based pattern. Each feature should have its own module (Controller, Service, optional DTO).
- **Dependency Injection**: Inject Mongoose Models into Services using `@InjectModel`.
  - **No Repository Pattern**: Do not create separate repository classes. Use Mongoose models directly within the service layer.

## 3. Database & ORM (Mongoose)

- **Model Mapping**:
  - Schemas should be defined in `schemas` folders within each module.
  - Collection names must be explicit in the `@Schema` decorator (e.g., `@Schema({ collection: 'users' })`).
- **Transactions & Locking**:
  - Business logic requiring atomicity must use Mongoose sessions/transactions (`session.startTransaction()`).
  - Use appropriate MongoDB atomic operations (e.g., `$inc`, `$set`) or optimistic concurrency control when necessary to prevent race conditions.
- **SKU Handling**:
  - SKUs are generated by the frontend and sent to the backend. The backend does not need to generate SKUs.

## 4. API & Error Handling

- **Errors**: Use standard NestJS `HttpException` (e.g., `NotFoundException`, `BadRequestException`) for error handling. Use `BusinessException` for business logic errors. Always write Vietnamese error messages.
- **Validation**:
  - Data Transfer Objects (DTOs) are **not mandatory** for every API but recommended for complex inputs.
- **Documentation**: Swagger decorators (`@ApiProperty`, etc.) are **not required** at this stage.

## 5. Workflow & AI Behavior

- **Imports**: Always ensure necessary modules and types are imported correctly.
- **Environment Variables**: Always ask the user for permission before reading or accessing values from `.env` files.
- **Dependencies**:
  - Before installing any new npm package/dependency, you **must explain why** it is needed and **ask for explicit approval** from the user.
- **Testing**: Unit tests are **not required** at this stage.

## 6. Project Context

- **Project Name**: DaoDuckWear (Clothing Shop).
- **Main Stack**: NestJS, Mongoose, MongoDB.

## 7. Database Seeding Rules

- **Supplemental Data**: When adding new seed data during development, **create a new separate seed file** (e.g., `src/database/seeders/seed-categories-v2.ts`) and run it. Do **not** run the main `seed.ts` if it contains `deleteMany()` operations that would wipe out current development data.
- **Master Seed Integrity**: Always update the main `seed.ts` file with the new data patterns. This ensures that the master seed remains the "source of truth" for fresh deployments or when other developers pull the repository.
