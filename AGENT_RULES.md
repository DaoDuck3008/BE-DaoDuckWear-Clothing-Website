# Backend Agent Rules - DaoDuckWear

Follow these rules strictly when developing the backend for DaoDuckWear.

## 1. General Standards
- **Language**: All rules, documentation, and code comments must be in **English**.
- **Naming Conventions**:
    - **Files**: Use `camelCase` (e.g., `productsService.ts`, `authController.ts`).
    - **Functions/Methods**: Use `camelCase` (e.g., `getProductById`, `createNewUser`).
    - **Classes**: Use `PascalCase` (e.g., `ProductsService`, `GlobalExceptionFilter`).
    - **Variables**: Use `camelCase`.

## 2. Project Architecture (NestJS)
- **Modular Structure**: Follow the existing module-based pattern. Each feature should have its own module (Controller, Service, optional DTO).
- **Dependency Injection**: Inject `PrismaService` directly into Services.
    - **No Repository Pattern**: Do not create separate repository classes. Use Prisma client directly within the service layer.

## 3. Database & ORM (Prisma)
- **Model Mapping**: 
    - ORM model names should be `camelCase` (or PascalCase as per Prisma defaults).
    - Database table names must be `snake_case` using the `@@map` attribute in `schema.prisma`.
- **Transactions & Locking**:
    - Business logic requiring atomicity must use `prisma.$transaction`.
    - Include **locking mechanisms** (e.g., Row-Level Locking via raw SQL or specific Prisma features) within transactions to prevent race conditions and conflicts, especially for inventory and order processing.
- **SKU Handling**: 
    - SKUs are generated by the frontend and sent to the backend. The backend does not need to generate SKUs.

## 4. API & Error Handling
- **Errors**: Use standard NestJS `HttpException` (e.g., `NotFoundException`, `BadRequestException`) for error handling.
- **Validation**: 
    - Data Transfer Objects (DTOs) are **not mandatory** for every API but recommended for complex inputs.
- **Documentation**: Swagger decorators (`@ApiProperty`, etc.) are **not required** at this stage.

## 5. Workflow & AI Behavior
- **Imports**: Always ensure necessary modules and types are imported correctly.
- **Environment Variables**: Always ask the user for permission before reading or accessing values from `.env` files.
- **Dependencies**: 
    - Before installing any new npm package/dependency, you **must explain why** it is needed and **ask for explicit approval** from the user.
- **Testing**: Unit tests are **not required** at this stage.

## 6. Project Context
- **Project Name**: DaoDuckWear (Clothing Shop).
- **Main Stack**: NestJS, Prisma, PostgreSQL.
